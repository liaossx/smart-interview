## 核心链路：6 个 Agent 流水线

### 整体流程 (graph.py)

```
                    START
                      │
                      ▼
              ┌─────────────────┐
              │   JD Analyzer   │  ← 把 JD 扔给 DeepSeek，提取技术栈、职责、级别
              └────────┬────────┘
                       │
                       ▼
              ┌───────────────────┐
              │  Resume Analyzer  │  ← 把简历文本扔给 DeepSeek，提取技能、项目、学历
              └────────┬──────────┘
                       │
                       ▼
              ┌──────────────────┐
              │   Gap Analyzer    │  ← 对比 JD 和简历，找出差距
              └────────┬─────────┘
                       │
                       ▼
              ┌─────────────────────┐
              │ Question Generator  │  ← 根据差距分析，生成 8-12 道面试题
              └────────┬────────────┘
                       │
                       ▼
              ┌──────────────────┐
              │    Interview      │  ← 逐题提问，LLM 评分，循环直到答完
              └────────┬─────────┘
                       │
             ┌─────────┴──────────┐
             │                    │
       还有下一题            全部答完
             │                    │
             ▼                    ▼
      回到 Interview        ┌──────────┐
                           │ Evaluator │  ← 综合所有答题记录，生成评估报告
                           └─────┬────┘
                                 │
                                 ▼
                                END
```

### 状态传递 (state.py)

所有 Agent 之间通过 `InterviewState`（TypedDict）传递数据：

```python
class InterviewState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]
    user_id: int
    session_id: str
    jd_content: str           # 输入的 JD
    resume_content: str        # 输入的简历文本
    jd_analysis: Dict          # JD 分析结果
    resume_analysis: Dict      # 简历分析结果
    gap_analysis: Dict         # 差距分析结果
    questions: List[Dict]      # 生成的面试题
    current_question_index: int
    answers: List[Dict]        # 答题记录
    evaluation: Dict           # 评估报告
    phase: str                 # 当前阶段
```

### 6 个 Agent 详解

#### 1. JD Analyzer (`jd_analyzer.py`)
- **输入**：JD 文本
- **输出**：结构化 JSON（tech_stack, responsibilities, required_skills, experience_level, role_type, summary）
- **方式**：System Prompt 要求 LLM 以 JSON 格式提取关键信息
- **痛点**：不同 JD 格式差异大，有些描述模糊，LLM 提取可能遗漏

#### 2. Resume Analyzer (`resume_analyzer.py`)
- **输入**：简历纯文本
- **输出**：结构化 JSON（skills, projects, education, work_experience, strengths, weaknesses）
- **方式**：System Prompt 要求 LLM 提取结构化信息
- **注意**：如果简历为空，跳过此节点

#### 3. Gap Analyzer (`gap_analyzer.py`)
- **输入**：JD 分析结果 + 简历分析结果
- **输出**：差距分析 JSON（matching_skills, jd_only_skills, interview_focus, difficulty）
- **作用**：找出"JD 要求但候选人不会"的技能，制定出题策略

#### 4. Question Generator (`question_generator.py`)
- **输入**：JD 分析 + 简历分析 + 差距分析
- **输出**：8-12 道面试题（含 category, question, purpose, difficulty, expected_answer_points）
- **分配**：40% 技术基础 + 30% 项目深挖 + 20% 场景设计 + 10% 软技能

#### 5. Interview Agent (`interview.py`)
- **方式**：ReAct Agent（create_react_agent），带 evaluate_answer 工具
- **流程**：逐题提问 → 接收回答 → 调用 evaluate_answer 评分 → 下一题
- **评分标准**：0-3 完全不懂 / 4-6 基础了解 / 7-8 掌握良好 / 9-10 深入理解

#### 6. Evaluator (`evaluator.py`)
- **输入**：所有问答记录 + JD/简历/差距分析
- **输出**：综合评估（overall_score, dimensions[], strengths[], weaknesses[], improvement_suggestions[], recommended_learning[]）
- **时机**：所有题目答完后执行

## 关键设计决策

### 为什么用 LangGraph 而不是直接调 API？

```
传统方式：                              LangGraph 方式：
function interview(jd, resume) {        jd_analyzer → resume_analyzer 
    let r1 = callLLM(jd)                → gap_analyzer → question_generator
    let r2 = callLLM(resume)            → interview(循环) → evaluator
    let r3 = callLLM(r1 + r2)
    // ... 全混在一起                每个节点独立，可单独修改 Prompt
}
```

**LangGraph 的优势**：
1. 每个 Agent 节点独立，修改某个 Prompt 不影响其他节点
2. 天然支持有条件循环（面试循环用 `should_continue_interview` 控制）
3. 状态通过 TypedDict 显式传递，流程清晰

### 状态管理

- 每次面试创建一个 `InterviewSession`（UUID 标识）
- 状态存储在 `InterviewSessionStore`（内存 Dict）
- **问题**：服务重启后 session 丢失
- **解决**：面试完成后将报告和 Q&A 写入后端 MySQL

### LLM 调用模式

- 每个 Agent 节点 = 一次 DeepSeek API 调用
- 一次 12 题面试 ≈ 14-15 次 API 调用
- System Prompt 控制输出格式（全部要求 JSON）
- 不同节点用不同 temperature：出题用 `get_creative_llm`（高 creativity），分析用 `get_fast_llm`（低 temperature）

用户上传 PDF/DOCX

    → PyMuPDF 提取 PDF 文本 或 python-docx 提取 DOCX 文本

## 面试官会问你的 5 个核心问题

### Q1: 为什么要用 LangGraph 而不是直接调 API？

> 直接用 API 调 DeepSeek 也能出题，但问题是一次面试涉及 JD 分析、简历分析、差距分析、出题、评分、综合评估等多个步骤。如果用普通代码写，这些逻辑会混在一起，每次修改都要改一大坨代码。LangGraph 把每个步骤做成独立节点，可以单独修改某个节点的 Prompt 而不影响其他节点，而且图结构一目了然。

### Q2: 这个图和普通的 if-else 链有什么区别？

> 面试的流程不是线性的——出完题后要循环面试，答完所有题才能评估。如果用传统的 if-else，状态管理会很混乱。LangGraph 的 StateGraph 天然支持这种有条件的循环，就是那个 `should_continue_interview` 函数判断是继续答题还是去评估。

### Q3: Agent 是怎么调用 LLM 的？每个请求都调一次 DeepSeek 吗？

> 对，每个 Agent 节点就是一次 DeepSeek API 调用。一次完整的 12 题面试大概要 14-15 次调用。这是目前最大的成本开销。优化方向有两个：一是把部分小任务（比如评分）用本地小模型替代，二是复用上下文减少重复传输。

### Q4: 简历解析你们是怎么做的？PDF 里的表格能正确解析吗？

> 目前是用 PyMuPDF 和 python-docx 提取纯文本，再把文本扔给 LLM 结构化提取。PDF 表格确实是个问题——纯文本提取会丢失表格结构。实际项目中如果要做得更好，可以用 Marker 或 LlamaParse 这类文档解析服务。

### Q5: 这个项目的难点是什么？你是怎么解决的？

准备 2-3 个你能讲清楚的技术难点：

- **Session 管理**：AI 服务是无状态的，每次面试的会话存在内存里，重启就丢了。后来加了 MySQL 持久化，面试完成后把报告和 Q&A 记录写回数据库。
- **大厂 vs 小厂出题策略**：小厂问实战能力，大厂问底层原理。同一个 Prompt 出所有题效果不好，需要根据公司类型动态调整 Prompt。
- **中文编码问题**：Windows 下 curl 发送中文请求会乱码，因为终端用 GBK 但后端用 UTF-8，踩了好几次坑。

## 学习路线建议

### 第一层：流程理解（1 小时）
对着 `graph.py` 里的 StateGraph 图，把 `jd_analyzer → resume_analyzer → gap_analyzer → question_generator → interview(循环) → evaluator` 这条线走一遍。能跟面试官讲清楚每个节点做什么就行。

### 第二层：关键概念（3 小时）
- 什么是 ReAct Agent（`create_react_agent`）
- StateGraph 的状态怎么在节点间传递（`InterviewState` TypedDict）
- System Prompt 如何控制 LLM 的输出格式（JSON）
- 认真读 `interview_service.py` 里的 `submit_answer()` — 这是评分+状态推进的核心

### 第三层：自己动手改一个东西（半天）
比如修改 `question_generator.py` 里的 SYSTEM_PROMPT，加上"如果公司是小厂，多出项目实战题；如果是大厂，多出底层原理题"的逻辑。然后做一次面试看看效果。这样面试官问起来你就能说"我亲手优化过出题策略"。

## 面试陈述建议

不要讲"我用 LangGraph 做了个 AI 面试系统"，而是讲：

> **我设计了一个多 Agent 协作的面试模拟系统。核心思路是把面试官的能力拆解成 6 个角色：JD 分析专家、简历分析专家、差距分析师、出题官、面试官、评估专家，每个角色由一个独立的 AI Agent 承担，用 LangGraph 编排它们协作。这样做的好处是每个 Agent 的 Prompt 可以单独优化，互不影响。**

## 项目优化方向

1. **大厂 vs 小厂出题策略**：JD Analyzer 识别公司规模 → Question Generator 按不同风格出题
2. **难度自适应**：根据前几题得分自动调整后续题目难度
3. **追问机制**：根据回答内容生成深入追问，模拟真实面试
4. **成本优化**：评分等简单任务用本地小模型替代 DeepSeek API
5. **文档解析增强**：支持 Markdown、表格、图片等更复杂的简历格式
